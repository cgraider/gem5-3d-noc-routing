🧱 مبانی gem5 و Ruby و Garnet

gem5 چیست؟

gem5 یک شبیه‌سازِ معماریِ کامپیوتر است؛ به‌جای اجرای برنامه روی سخت‌افزارِ واقعی، رفتارِ آن را شبیه‌سازی می‌کند تا بتوان آن را به‌دقت اندازه‌گیری و بررسی کرد.

🟣 مهم

gem5 رویداد-محور است؛ یعنی هیچ بخشی در یک حلقهٔ دائمی اجرا نمی‌شود. هر مؤلفه برای زمانی مشخص یک رویداد ثبت می‌کند و در آن زمان، تابعِ wakeup() آن فراخوانی می‌شود. کلِ شبیه‌سازی، اجرای پشتِ‌سرِهمِ همین رویدادها روی یک خطِ زمانی است.

gem5  (موتورِ شبیه‌سازی، ++C)
  └─ Ruby      ← مدلِ دقیقِ حافظه و شبکه
       └─ Garnet   ← مدلِ شبکه‌روی‌تراشه (NoC) — کدِ این پروژه اینجاست
            └─ Router → InputUnit / RoutingUnit / SwitchAllocator / CrossbarSwitch / OutputUnit

بسته، فلیت و کانالِ مجازی

  • بسته (Packet): واحدِ منطقیِ داده که از مبدأ به مقصد ارسال می‌شود.
  • فلیت (Flit): هر بسته به چند فلیت تقسیم می‌شود؛ یک فلیتِ سَر (HEAD) که مسیر و زمان‌ها را همراه دارد، چند فلیتِ بدنه (BODY) و یک فلیتِ دُم (TAIL) که پایانِ بسته را مشخص می‌کند.
  • کانالِ مجازی (VC): هر لینکِ فیزیکی به چند صفِ بافرِ مستقل تقسیم می‌شود که از گیر کردنِ صف جلوگیری می‌کنند.
  • اعتبار (Credit): نشان‌دهندهٔ فضای خالیِ بافر در سمتِ مقصد است. روتر تنها زمانی فلیت می‌فرستد که اعتبار داشته باشد.

🟣 مهم

تصمیمِ مسیریابی برای هر بسته یک‌بار و روی فلیتِ سَر گرفته می‌شود و همان پورتِ خروجی برای بقیهٔ فلیت‌های همان بسته به کار می‌رود. پس هر تصمیمِ مسیریابی معادلِ یک پرش (hop) است.

🟢 ترفند

میزانِ اعتبار یا فضای خالیِ بافر، همان سیگنالِ شلوغی (congestion) است. هر چهار الگوریتمِ این پروژه به‌نوعی همین مقدار را می‌خوانند تا شلوغیِ هر جهت را تشخیص دهند.

خط‌لولهٔ روتر (یک پرش)

زیرواحدوظیفه
InputUnitفلیتِ ورودی را از لینک می‌گیرد و در بافرِ کانالِ مجازی قرار می‌دهد.
RoutingUnitپورتِ خروجی را تعیین می‌کند. کدِ مسیریابیِ این پروژه در همین زیرواحد است.
SwitchAllocatorتعیین می‌کند کدام فلیت در این سیکل از کراس‌بار عبور کند.
CrossbarSwitchفلیتِ برنده را به پورتِ خروجی منتقل می‌کند.
OutputUnitفلیت را روی لینک می‌فرستد و اعتبارها را مدیریت می‌کند.

RoutingUnit و محلِ اتصالِ الگوریتم‌ها

تابعِ outportCompute() شمارهٔ الگوریتم را از آرگومانِ --routing-algorithm=N می‌خواند و اجرای متناظر را انتخاب می‌کند:

switch (routing_algorithm) {
case TABLE_:    outport = lookupRoutingTable(...);    break; // 0
case XY_:       outport = outportComputeXY(...);       break; // 1
case DEEPNR3D_: outport = outportComputeDeepNR3D(...); break; // 2
case PROPOSED_: outport = outportComputeProposed(...); break; // 3
case XYZ_:      outport = outportComputeXYZ(...);      break; // 4
case CAQR_:     outport = outportComputeCAQR(...);     break; // 5
}
🔧 یادآوریِ فنی — enum و switch

نام‌هایی مانندِ TABLE_ و XY_ مقادیرِ یک enum هستند؛ یعنی نام‌هایی برای اعدادِ ۰ تا ۵. دستورِ switch نیز بسته به مقدارِ متغیر، یکی از شاخه‌ها را اجرا می‌کند.

وظیفهٔ هر تابعِ مسیریابی روشن است: یک عدد بازمی‌گرداند که همان شمارهٔ پورتِ خروجی است. برای این کار از دو قراردادِ زیر استفاده می‌شود.

نگاشتِ جهت به شمارهٔ پورت

m_outports_dirn2idx["North"]  // شمارهٔ پورتِ خروجی به‌سمتِ همسایهٔ شمالی
m_inports_dirn2idx["South"]   // شمارهٔ پورتِ ورودی‌ای که فلیت از آن وارد شده

شش جهتِ سه‌بعدی عبارت‌اند از: North, East, South, West, Up, Down. جهت‌های Up و Down همان لینک‌های عمودیِ TSV بینِ لایه‌ها هستند.

🔧 یادآوریِ فنی — std::map

std::map<Key, Value> یک ساختارِ کلید-مقدار (دیکشنری) است؛ با یک کلید مانندِ "North" (یک string) مقدارِ متناظر (مثلاً شمارهٔ پورت) را برمی‌گرداند.

رابطهٔ شمارهٔ روتر با مختصات (x, y, z)

id = z * (rows * cols) + y * cols + x

z = id / (rows*cols)
y = (id % (rows*cols)) / cols
x = (id % (rows*cols)) % cols

🟣 مهم

این رابطه در هر چهار الگوریتم به کار می‌رود و پلِ میانِ شمارهٔ خطیِ روتر و هندسهٔ شبکه است (محاسبهٔ فاصلهٔ منهتن، تشخیصِ لبه‌ها و انتخابِ جهتِ نزدیک‌شونده به مقصد).

🔴 هشدار

خطا در این رابطه باعثِ مسیریابیِ نادرستِ همهٔ الگوریتم‌ها می‌شود؛ پس باید با دقت رعایت شود.

🔧 یادآوریِ فنی — عملگرهای / و %

روی اعدادِ صحیح، / تقسیمِ صحیح است (باقی‌مانده حذف می‌شود) و % باقی‌ماندهٔ تقسیم را می‌دهد. برای مثال با cols=4 و id=6 داریم: x = 6 % 4 = 2 و y = 6 / 4 = 1.

خواندنِ شلوغی از OutputUnit

// فضای خالیِ بافر (بیشتر = خلوت‌تر) — در DeepNR3D و Proposed استفاده می‌شود:
m_router->getOutputUnit(port)->get_credit_count(vc);

// عمقِ صفِ خروجی (بیشتر = شلوغ‌تر) — در CAQR استفاده می‌شود:
m_router->getOutputUnit(port)->getOutQueue()->getSize();

قانونِ اندازهٔ شبکه

🟣 مهم

num-cpus == num-dirs == rows × cols × layers. برای مثال شبکهٔ ۴×۴×۲ برابرِ ۳۲ گره است.

حلقهٔ یادگیریِ تقویتی (تنها در DeepNR3D و Proposed)

RoutingUnit در gem5 (++C، سمتِ کلاینت ZMQ)  ──حالت──►  agent.py (پایتون، سمتِ سرور)
                                            ◄──اکشن──

🔵 نکته

فهمِ سمتِ gem5 نیازی به دانشِ یادگیریِ ماشین ندارد؛ این بخش تنها حالت را ارسال و پورتِ بازگشتی را اعمال می‌کند. کلِ منطقِ یادگیری در عاملِ پایتونی قرار دارد.

🎬 مسیرِ اجرای شبیه‌سازی

این بخش ترتیبِ رویدادها را از تزریقِ یک بسته تا رسیدنِ آن به مقصد دنبال می‌کند.

اسکریپتِ اجرا (shell)
  └─ باینریِ gem5 (موتورِ ++C)
       └─ اسکریپتِ کانفیگ (پایتون)
            └─ شبکهٔ Ruby / Garnet (++C)
                 └─ RoutingUnit ──ZMQ──► agent.py (تنها در DeepNR3D و Proposed)
  └─ آمار در stats.txt نوشته می‌شود
🔧 یادآوریِ فنی — ارتباطِ پایتون و ++C

پایتون و ++C در یک پروسهٔ مشترک اجرا می‌شوند. به همین دلیل کدِ ++C می‌تواند با std::getenv("GARNET_ROUTING_ALGORITHM") همان متغیرِ محیطی‌ای را بخواند که اسکریپتِ پایتون با os.environ[...] تنظیم کرده است. این روشِ سادهٔ تبادلِ اطلاعات میانِ دو زبان است.

تصمیمِ مسیریابی

در الگوریتم‌های یادگیرنده، در هر پرش این مراحل اجرا می‌شود:

۱. دریافتِ پاداشِ پرشِ قبلی          DeepNR::get_reward(packet_id)
۲. ساختِ بردارِ حالت                 [one-hot روترِ فعلی و مقصد، فاصله، وضعیتِ بافرها]
۳. تعیینِ اکشن‌های مجاز               (جهتِ خارج از لبه یا با بافرِ پر، مجاز نیست)
۴. ارسالِ حالت با JSON (ZMQ)          → agent.py
۵. انتظار و دریافتِ اکشن              ← agent.py
۶. اعتبارسنجیِ اکشن                  در صورتِ نامعتبر بودن → پایانِ اپیزود
۷. نگاشتِ اکشن به پورتِ خروجی و بازگرداندن

ذخیرهٔ پاداش (OutputUnit.cc)

float reward = 1.0f / (queuing_cycles + 1.0f);  // تأخیرِ کم → پاداشِ نزدیک به ۱
DeepNR::store_reward(packet_id, reward, queuing_cycles);

🔵 نکته

پاداش با یک پرش تأخیر محاسبه می‌شود، زیرا نتیجهٔ یک اکشن تا رسیدنِ بسته به روترِ بعدی مشخص نمی‌شود. همین الگو در CAQR نیز وجود دارد.

🧭 الگوریتم XYZ ۴

XYZ یک مسیریابیِ قطعیِ سه‌بعدی است؛ بدونِ عامل، بدونِ یادگیری و بدونِ حالت، تنها یک قاعدهٔ ثابت. توپولوژیِ آن Mesh_3D است.

🟢 ترفند

XYZ نقشِ خطِ مبنا (baseline) را دارد؛ مرجعی که الگوریتم‌های پیچیده‌تر با آن مقایسه می‌شوند. ساده است اما همیشه درست کار می‌کند.

ایدهٔ اصلی

ابعاد همیشه به ترتیبِ ثابت حل می‌شوند: ابتدا X، سپس Y و در پایان Z. چون همهٔ بسته‌ها همین ترتیب را رعایت می‌کنند، شبکه بدونِ بن‌بست (deadlock-free) باقی می‌ماند.

🟣 مهم

ضعفِ اصلیِ XYZ این است که شلوغی را در نظر نمی‌گیرد. همین محدودیت، فضایی است که CAQR و الگوریتم‌های یادگیرنده برای بهبودِ آن طراحی شده‌اند.

بخشِ کلیدیِ کد

if      (dest_x > my_x) outport_dirn = "East";
else if (dest_x < my_x) outport_dirn = "West";
else if (dest_y > my_y) outport_dirn = "South";
else if (dest_y < my_y) outport_dirn = "North";
else if (dest_z > my_z) outport_dirn = "Up";    // ‎+Z با TSV
else if (dest_z < my_z) outport_dirn = "Down";  // ‎-Z با TSV
else  panic("XYZ: مبدأ برابرِ مقصد");

return m_outports_dirn2idx[outport_dirn];
🔧 یادآوریِ فنی — زنجیرهٔ if / else if

این زنجیره خودِ «ترتیبِ ابعاد» را پیاده می‌کند؛ با برقرار شدنِ یک شرط، بقیه نادیده گرفته می‌شوند. تا زمانی که X هم‌تراز نشده، نوبت به Y نمی‌رسد و تا Y هم‌تراز نشده، نوبت به Z نمی‌رسد. عبارتِ panic(...) نیز حالتی را علامت می‌زند که نباید رخ دهد و در صورتِ وقوع، اجرا را با خطا متوقف می‌کند.

اجرا

./build/ALL/gem5.opt configs/example/garnet_synth_traffic.py \
    --network=garnet --num-cpus=32 --num-dirs=32 \
    --topology=Mesh_3D --mesh-rows=4 --mesh-layers=2 --vcs-per-vnet=2 \
    --routing-algorithm=4 --link-latency=1 --router-latency=1 \
    --sim-cycles=1000000 --synthetic=uniform_random --injectionrate=0.1

🧠 الگوریتم CAQR ۵

CAQR یا «مسیریابیِ Qِ آگاه از شلوغی»، یک یادگیریِ تقویتیِ جدولی روی شبکهٔ دوبعدی (Mesh_XY) است. این الگوریتم به‌طورِ کامل در ++C پیاده شده و به عاملِ پایتونی نیاز ندارد.

ایدهٔ اصلی

یک جدولِ Q نگه‌داری می‌شود که به این پرسش پاسخ می‌دهد: «از روترِ x و با مقصدِ d، حرکت از جهتِ dir چه هزینه‌ای دارد؟». هرچه مقدارِ Q کمتر باشد، آن جهت بهتر است و روتر همواره جهتِ با کمترین Q را انتخاب می‌کند.

namespace CAQR {
// qtable[router][dest][dir] = مقدارِ Q
static std::map<int, std::map<int, std::map<std::string, double>>> qtable;
static const int    TRAIN_STEPS = 50;  // طولِ فازِ اکتشاف
static const double ALPHA = 0.5;       // نرخِ یادگیری α
static const double GAMMA = 0.7;       // نرخِ تخفیف γ
static const double EPSILON = 0.5;     // احتمالِ اکتشاف ε
}
🔧 یادآوریِ فنی — static و map های تو‌در‌تو

کلیدواژهٔ static سبب می‌شود متغیر میانِ فراخوانی‌های متوالیِ تابع پایدار بماند و میانِ همهٔ روترها مشترک باشد؛ همین حافظه است که یادگیری را ممکن می‌کند. بدونِ آن، جدول در هر فراخوانی از نو خالی می‌شد.

std::map<int, std::map<int, std::map<string, double>>> یک ساختارِ سه‌سطحی است؛ برای نمونه qtable[5][20]["East"] هزینهٔ حرکت از روترِ ۵ به‌سمتِ مقصدِ ۲۰ از جهتِ شرق را نگه می‌دارد.

انتخابِ جهت: از اکتشاف تا بهره‌برداری

bool exploring = (CAQR::packet_count < CAQR::TRAIN_STEPS) &&
                 ((double)rand() / RAND_MAX < CAQR::EPSILON);

if (exploring || feasible.size() == 1) {
    chosen_dirn = feasible[rand() % feasible.size()];   // انتخابِ تصادفی
} else {
    chosen_dirn  = feasible[0];                          // انتخابِ کمترین Q
    double min_q = CAQR::getQ(my_id, dest_id, feasible[0]);
    for (size_t i = 1; i < feasible.size(); ++i) {
        double q = CAQR::getQ(my_id, dest_id, feasible[i]);
        if (q < min_q) { min_q = q; chosen_dirn = feasible[i]; }
    }
}

🟢 ترفند

این همان موازنهٔ شناخته‌شدهٔ اکتشاف در برابرِ بهره‌برداری است: در ابتدا مسیرهای تازه آزموده می‌شوند تا جدول شکل بگیرد و پس از آن، بهترین مسیرِ آموخته‌شده به کار می‌رود.

به‌روزرسانیِ Q (رابطهٔ مقاله)

Q_x(y,d)_new = Q_x(y,d)_old + α · ( γ · Q_y(z,d) + q_y + δ_xy − Q_x(y,d)_old )
نمادتوضیح
q_yسیگنالِ شلوغی؛ عمقِ صفِ خروجی در روترِ بعدی (همان وجهِ «آگاه از شلوغی»)
δ_xyتأخیرِ ثابتِ لینک، برابرِ ۱ سیکل
α, γنرخِ یادگیری ۰٫۵ و نرخِ تخفیف ۰٫۷

🟣 مهم — چرا یک پرش تأخیر؟

هزینهٔ واقعیِ یک پرش تا رسیدنِ بسته به روترِ بعدی مشخص نیست، چون باید عمقِ صفِ آنجا خوانده شود. به همین دلیل هر روتر مقدارِ Qِ روترِ قبلی را بر اساسِ شلوغیِ محلیِ خود به‌روزرسانی می‌کند.

🔧 یادآوریِ فنی — تفسیرِ رابطه

ساختارِ Q_new = Q_old + α · (هدف − Q_old) یعنی مقدارِ قبلیِ Q به‌اندازهٔ ضریبِ α (اینجا نصف) به‌سمتِ یک مقدارِ هدف نزدیک می‌شود. این همان قاعدهٔ کلاسیکِ Q-learning است که با افزودنِ ترمِ شلوغیِ q_y به آگاهی از شلوغی مجهز شده است.

🤖 الگوریتم DeepNR3D ۲

DeepNR3D یک شبکهٔ Qِ عمیق (DQN) است که مسیریابی را می‌آموزد. توپولوژیِ آن Mesh_3D و پورتِ ZMQِ آن ۵۵۵۵ است.

🔵 نکته

تفاوتِ اصلی با CAQR این است که اینجا به‌جای جدول، یک شبکهٔ عصبی از روی بردارِ حالت برای هر جهت یک مقدارِ Q پیش‌بینی می‌کند و این شبکه در یک برنامهٔ پایتونیِ جداگانه اجرا می‌شود.

🔧 یادآوریِ فنی — ZMQ و الگوی REQ/REP

ZMQ کتابخانه‌ای برای پیام‌رسانی میانِ دو برنامه است. در الگوی درخواست/پاسخ، یک طرف درخواست می‌فرستد و منتظر می‌ماند و طرفِ دیگر پاسخ می‌دهد. در اینجا gem5 نقشِ کلاینت و عاملِ پایتونی نقشِ سرور را دارد؛ چون gem5 منتظر می‌ماند، شبیه‌سازی تا دریافتِ پاسخ متوقف می‌شود.

بردارِ حالت (طول = 2·num_routers + 8)

ویژگیتوضیح

🔧 یادآوریِ فنی — رمزگذاریِ one-hot

در رمزگذاریِ one-hot تنها یک درایهٔ بردار برابرِ ۱ و بقیه ۰ است؛ برای نمونه در شبکه‌ای با ۴ روتر، اگر روترِ فعلی شمارهٔ ۲ باشد بردار برابرِ [0, 0, 1, 0] می‌شود. این روشِ متداولِ نمایشِ «کدام مورد از میانِ چند مورد» برای شبکهٔ عصبی است.

ماسکِ اکشن‌های مجاز

std::vector<bool> available_actions(6, true);
if (my_y == 0)            available_actions[0] = false; // فاقدِ همسایهٔ شمالی (لبه)
if (my_x == num_cols - 1) available_actions[1] = false; // فاقدِ همسایهٔ شرقی
// در صورتِ پر بودنِ بافرِ یک جهت نیز آن جهت غیرمجاز می‌شود

🔴 هشدار

در صورتِ دریافتِ اکشنِ نامعتبر، gem5 به‌عمد اپیزود را با fatal() پایان می‌دهد. این رفتار خطا نیست؛ علامتی برای حلقهٔ آموزش است تا اپیزودِ بعدی را آغاز کند.

سمتِ پایتون — معماریِ شبکهٔ عصبی

state_size → 64 (فشرده‌سازی) → ReLU
64 → 256 → 128 → 64 → action_size (۶ خروجی؛ مقدارِ Q برای هر جهت)
🔧 یادآوریِ فنی — epsilon-greedy و بافرِ تجربه

در سیاستِ epsilon-greedy با احتمالِ epsilon یک جهتِ تصادفیِ مجاز (اکتشاف) و در غیرِ این صورت جهتِ با بیشترین Q (بهره‌برداری) انتخاب می‌شود؛ مقدارِ epsilon به‌تدریج از ۰٫۹ به ۰٫۰۱ کاهش می‌یابد.

بافرِ تجربه (ReplayBuffer) چهارتایی‌های (s, a, r, s', done) را نگه می‌دارد و هنگامِ آموزش، یک دستهٔ تصادفی از آن انتخاب می‌شود تا همبستگیِ نمونه‌های متوالی، آموزش را منحرف نکند.

اجرا (دو ترمینال)

# ترمینالِ ۱ — عامل (۳۲ روتر → state = 2*32+8 = 72)

⭐ روشِ پیشنهادی (Proposed) ۳

افزودهٔ اصلی: حافظهٔ EMA برای شلوغی

static std::map<int, std::array<float, 6>> ema_occ;  // برای هر روتر، شش جهت
static const float EMA_ALPHA = 0.1f;

ema_occ[my_id][d] = EMA_ALPHA * buf_occ[d] + (1.0f - EMA_ALPHA) * ema_occ[my_id][d];
🔧 یادآوریِ فنی — میانگینِ متحرکِ نمایی (EMA)

EMA یک میانگینِ هموار است که سابقهٔ مقادیر را نیز در نظر می‌گیرد. با α=0.1 هر اندازه‌گیریِ تازه تنها ۱۰٪ در نتیجه اثر می‌گذارد، در نتیجه تخمینی آرام و کم‌نوسان از شلوغی به دست می‌آید؛ این به عامل دیدی تاریخی از شلوغیِ هر جهت می‌دهد، نه صرفاً مقدارِ لحظه‌ای.

نگاشتِ اکشن با جدولِ delta

static const int delta[6][3] = {{0,-1,0},{1,0,0},{0,1,0},{-1,0,0},{0,0,1},{0,0,-1}};
int nx = my_x + delta[action][0];
int ny = my_y + delta[action][1];
int nz = my_z + delta[action][2];
int next_id = nz * layer_size + ny * num_cols + nx;
🔧 یادآوریِ فنی — کاربردِ جدولِ delta

به‌جای زنجیرهٔ طولانیِ if/else برای شش جهت، یک جدولِ کوچک هر اکشن (۰ تا ۵) را به تغییرِ مختصاتِ (dx, dy, dz) نگاشت می‌کند. نتیجه کوتاه‌تر، خواناتر و کم‌خطاتر است.

تابعِ خطای Huber به‌جای MSE

loss = nn.functional.smooth_l1_loss(current_q.squeeze(), target_q)
🔧 یادآوریِ فنی — MSE در برابرِ Huber

تابعِ MSE خطاهای بزرگ را با توانِ دو جریمه می‌کند و یک پاداشِ پرت می‌تواند آموزش را ناپایدار کند. تابعِ Huber برای خطاهای کوچک مانندِ MSE و برای خطاهای بزرگ به‌صورتِ خطی عمل می‌کند؛ از این رو در شبکه‌ای با پاداش‌های پرنوسان، آموزش را پایدارتر می‌سازد.

🟣 جمع‌بندی

Proposed را می‌توان همان DeepNR3D دانست که با بردارِ حالتِ غنی‌تر (شاملِ دیدِ تاریخی از شلوغی) و آموزشِ پایدارتر تقویت شده است.

🔴 هشدارِ مهم

gem5 روی ویندوز ساخته و اجرا نمی‌شود؛ مسیرِ v:\... تنها یک درایوِ متصل (mount) است. همهٔ دستورهای ساخت و اجرا باید روی سرورِ لینوکس اجرا شوند.

# فعال‌سازیِ محیط
source ~/gem5-env/bin/activate
cd ~/gem5_new/gem5/gem5

# نسخهٔ بهینه (نسخهٔ معمولِ اجرا)
scons build/ALL/gem5.opt -j$(nproc)

# نسخهٔ دیباگ (کندتر، با پیام‌های خطای دقیق‌تر)
scons build/ALL/gem5.debug -j$(nproc)
🔧 یادآوریِ فنی — scons و ‎-j$(nproc)

scons ابزارِ ساختِ gem5 است (مشابهِ make) و تنها فایل‌های تغییریافته را بازکامپایل می‌کند. گزینهٔ -j$(nproc) از همهٔ هسته‌های پردازنده استفاده می‌کند. ساختِ نخست بین ۲۰ تا ۶۰ دقیقه و ساخت‌های بعدی چند ثانیه طول می‌کشد.

🟣 مهم — وابستگیِ ZMQ

برای DeepNR3D و Proposed باید بستهٔ libzmq3-dev نصب باشد. در نبودِ آن، کامپایل موفق می‌شود اما این دو الگوریتم هنگامِ اجرا fatal() می‌دهند، چون کدشان درونِ #ifdef USE_ZMQ قرار دارد.
