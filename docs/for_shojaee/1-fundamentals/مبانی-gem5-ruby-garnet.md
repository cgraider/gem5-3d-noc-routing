## ۱) gem5 اصلاً چیست؟

**gem5** یک **شبیه‌ساز کامپیوتر** است. یعنی به‌جای اینکه روی یک سخت‌افزار واقعی برنامه اجرا
کنیم، gem5 رفتار آن سخت‌افزار را با نرم‌افزار **شبیه‌سازی** می‌کند تا بتوانیم اندازه‌گیری و
آزمایش کنیم.

> [!IMPORTANT]
> 🟣 gem5 **رویداد-محور (event-driven)** است. یعنی هیچ‌چیز توی یک حلقه‌ی بی‌نهایت نمی‌چرخد؛
> هر قطعه (مثل یک روتر) یک «رویداد» برای یک زمان مشخص ثبت می‌کند و وقتی آن زمان رسید، تابعِ
> `wakeup()` آن قطعه صدا زده می‌شود. کل شبیه‌سازی = پشتِ‌سرِ‌هم اجرا شدنِ این رویدادها روی یک
> خطِ زمان (tick).

ساختار لایه‌ای که با آن کار داریم این است:

```
gem5  (موتور شبیه‌سازی، زبان C++)
  └─ Ruby      ← مدل دقیق حافظه و شبکه‌ی به‌هم‌پیوسته
       └─ Garnet   ← مدلِ «شبکه‌روی‌تراشه» (NoC). کدِ ما اینجاست! 🎯
            └─ Router → InputUnit / RoutingUnit / SwitchAllocator / CrossbarSwitch / OutputUnit
```

- **Ruby**: مدل حافظه‌ی دقیق gem5. ما با بخش حافظه‌اش کاری نداریم، فقط ازش برای ساختنِ شبکه
  استفاده می‌کنیم.
- **Garnet**: مدلِ شبکه‌ی داخلِ Ruby. **تمام کدِ مسیریابیِ ما اینجاست:**
  `src/mem/ruby/network/garnet/`

> [!NOTE]
> 🔵 ما شبکه را در حالت **ترافیک مصنوعی (synthetic traffic)** اجرا می‌کنیم. یعنی به‌جای
> ترافیک واقعی پردازنده، یک تولیدکننده‌ی الگو بسته‌ها را با یک الگوی مشخص (مثلاً تصادفیِ
> یکنواخت `uniform_random`) تزریق می‌کند. این‌طوری شبکه را جدا می‌کنیم تا فقط **مسیریابی** را
> بسنجیم.

## ۲) بسته، فلیت و کانال مجازی (مهم‌ترین واژه‌ها)

- **Packet (بسته):** واحد منطقی‌ای که از یک گره‌ی مبدأ به یک گره‌ی مقصد فرستاده می‌شود.
- **Flit (فلیت):** بسته به تکه‌های کوچک‌تری به نام فلیت شکسته می‌شود:
  - یک فلیتِ **HEAD** (سَر) — مسیر و زمان‌ها را با خودش دارد،
  - چند فلیتِ **BODY** (بدنه)،
  - یک فلیتِ **TAIL** (دُم) — پایان بسته را مشخص می‌کند.
  - اگر بسته فقط یک فلیت باشد، می‌شود **HEAD_TAIL**.
- **Virtual Channel / VC (کانال مجازی):** یک لینکِ فیزیکی به چند صف بافرِ مستقل تقسیم می‌شود
  که روی همان سیم با هم شریک‌اند. VC جلوی گیر کردنِ صف (head-of-line blocking) را می‌گیرد.
- **Credit (اعتبار):** نشان می‌دهد در سمتِ مقصد چقدر جای خالیِ بافر هست. روتر فقط وقتی
  می‌تواند فلیت بفرستد که **اعتبار** داشته باشد.

> [!IMPORTANT]
> 🟣 این جمله را حفظ کن: **تصمیمِ مسیریابی برای هر بسته، یک‌بار و سرِ فلیتِ HEAD گرفته می‌شود.**
> پورتِ خروجی‌ای که انتخاب شد، برای فلیت‌های BODY و TAILِ همان بسته دوباره استفاده می‌شود.
> پس وقتی می‌گوییم «تصمیم مسیریابی»، یعنی سرِ هر **پرش (hop)** برای فلیتِ HEAD.

> [!TIP]
> 🟢 **مقدار اعتبار / جای خالی بافر = سیگنالِ شلوغی (congestion).** همه‌ی الگوریتم‌های این
> پروژه به یک شکلی همین را می‌خوانند تا بفهمند کدام جهت شلوغ است.

## ۳) خط‌لوله‌ی روتر (یک پرش)

هر بار که `Router::wakeup()` صدا زده می‌شود، این زیرواحدها کار می‌کنند:

| زیرواحد | کارش |
|---|---|
| **InputUnit** | فلیتِ ورودی را از لینک می‌گیرد و در بافرِ VC می‌گذارد. |
| **RoutingUnit** | برای فلیتِ HEAD، **پورتِ خروجی را حساب می‌کند.** ← 🎯 کدِ ما اینجاست |
| **SwitchAllocator** | داوری می‌کند کدام فلیت این سیکل از کراس‌بار رد شود. |
| **CrossbarSwitch** | فلیتِ برنده را به پورتِ خروجی منتقل می‌کند. |
| **OutputUnit** | صفِ خروجی را دارد، فلیت را روی لینک می‌فرستد و اعتبارها را می‌شمارد. |

> [!NOTE]
> 🔵 ما فقط **RoutingUnit** را دست می‌زنیم. بقیه‌ی زیرواحدها همان‌طور که هستند می‌مانند.

## ۴) RoutingUnit — همه‌ی الگوریتم‌ها اینجا وصل می‌شوند

فایل: `src/mem/ruby/network/garnet/RoutingUnit.cc`

تابع `outportCompute()` نقطه‌ی تقسیم (dispatch) است. شماره‌ی الگوریتم را از روی
`--routing-algorithm=N` می‌خواند و به پیاده‌سازیِ درست سوییچ می‌کند:

```cpp
switch (routing_algorithm) {
case TABLE_:    outport = lookupRoutingTable(...);    break; // 0
case XY_:       outport = outportComputeXY(...);       break; // 1
case DEEPNR3D_: outport = outportComputeDeepNR3D(...); break; // 2
case PROPOSED_: outport = outportComputeProposed(...); break; // 3
case XYZ_:      outport = outportComputeXYZ(...);      break; // 4
case CAQR_:     outport = outportComputeCAQR(...);     break; // 5
}
```

> 🎓 **آموزش کوچک C++ — `enum` و `switch`**
> آن `TABLE_`, `XY_`, ... یک **`enum`** هستند؛ یعنی فقط اسم‌های قشنگ برای عددها (۰، ۱، ۲، ...).
> تعریفشان در فایل `CommonTypes.hh` است:
> ```cpp
> enum RoutingAlgorithm { TABLE_=0, XY_=1, DEEPNR3D_=2, PROPOSED_=3, XYZ_=4, CAQR_=5, ... };
> ```
> `switch` هم یعنی «بسته به مقدارِ این متغیر، یکی از این شاخه‌ها را اجرا کن». ساده ولی پرکاربرد.

**وظیفه‌ی هر تابعِ مسیریابی فقط یک چیز است: یک عدد برگرداند → شماره‌ی پورتِ خروجی (`int`).**
برای این کار با دو قرارداد مهمِ پروژه کار می‌کند:

### ۴.۱) رشته‌ی جهت ↔ شماره‌ی پورت

هر روتر یک map دارد که اسمِ جهت را به شماره‌ی پورتِ فیزیکی ترجمه می‌کند:

```cpp
m_outports_dirn2idx["North"]  // شماره‌ی پورتِ خروجی به‌سمتِ همسایه‌ی شمالی
m_inports_dirn2idx["South"]   // شماره‌ی پورتِ ورودی‌ای که فلیت از آن آمده
```

شش جهتِ سه‌بعدی: `North, East, South, West, Up, Down`. جهتِ `Up`/`Down` همان لینک‌های عمودیِ
**TSV** بینِ لایه‌ها هستند (فقط در توپولوژیِ سه‌بعدی `Mesh_3D` وجود دارند).

> 🎓 **آموزش کوچک C++ — `std::map`**
> `std::map<KeyType, ValueType>` یک «دیکشنری» است: با یک کلید (مثلاً رشته‌ی `"North"`) یک
> مقدار (مثلاً عدد پورت) می‌گیری. نوشتنِ `m_outports_dirn2idx["North"]` یعنی «مقدارِ متناظرِ
> کلیدِ `North` را بده».

### ۴.۲) شماره‌ی روتر ↔ مختصات (x, y, z) — 🟣 این را خوب یاد بگیر

شماره‌ی روترها سطر‌به‌سطر و لایه‌به‌لایه شماره‌گذاری شده‌اند:

```
id = z * (rows * cols) + y * cols + x

z = id / (rows*cols)
y = (id % (rows*cols)) / cols
x = (id % (rows*cols)) % cols
```

> [!IMPORTANT]
> 🟣 این فرمول در **همه‌ی** الگوریتم‌ها (XYZ، CAQR، DeepNR3D، Proposed) ظاهر می‌شود. این پل
> بینِ شماره‌ی صافِ روتر و هندسه‌ی شبکه است (فاصله‌ی منهتن، اینکه روی لبه هستیم یا نه، کدام
> جهت ما را به مقصد نزدیک‌تر می‌کند).

> [!WARNING]
> 🔴 اگر این فرمول را اشتباه بنویسی، **همه‌ی** الگوریتم‌ها غلط مسیریابی می‌کنند. خیلی مراقبش باش.

برای شبکه‌ی دوبعدی (`Mesh_XY`) چون `layers = 1` است، فرمول ساده می‌شود به: `id = y*cols + x`.

> 🎓 **آموزش کوچک C++ — `/` و `%`**
> در C++ روی عددهای صحیح، `/` تقسیمِ صحیح است (باقی‌مانده را دور می‌ریزد) و `%` باقی‌ماندهٔ
> تقسیم است. مثلاً اگر `cols=4` و `id=6` باشد: `x = 6 % 4 = 2` و `y = 6 / 4 = 1`. یعنی روترِ
> شماره‌ی ۶ در ستونِ ۲ و سطرِ ۱ است.

## ۵) خواندنِ شلوغی: اعتبار و صفِ OutputUnit

تصمیمِ مسیریابی به این بستگی دارد که هر جهتِ خروجی چقدر شلوغ است. دو منبع داریم:

- **اعتبارِ خالی (جای خالیِ بافرِ پایین‌دست)** — هرچه بیشتر، **کم‌شلوغ‌تر**:
  ```cpp
  m_router->getOutputUnit(port)->get_credit_count(vc);
  ```
  DeepNR3D و Proposed این را نرمال می‌کنند و توی بردارِ حالت می‌گذارند.
- **عمقِ صفِ خروجی (تعداد فلیت‌های منتظر)** — هرچه بیشتر، **شلوغ‌تر**:
  ```cpp
  m_router->getOutputUnit(port)->getOutQueue()->getSize();
  ```
  CAQR مستقیماً از همین به‌عنوانِ «تأخیرِ صف» (`q_y`) در فرمولش استفاده می‌کند.

## ۶) توپولوژی: شبکه چطور ساخته می‌شود

یک اسکریپتِ پایتونی قبل از شبیه‌سازی روترها و لینک‌ها را سیم‌کشی می‌کند:

- `configs/topologies/Mesh_XY.py` — شبکه‌ی دوبعدیِ استاندارد.
- `configs/topologies/Mesh_3D.py` — شبکه‌ی سه‌بعدیِ سفارشی با لینک‌های عمودیِ TSV.

> [!IMPORTANT]
> 🟣 **قانونِ اندازه:** `num-cpus == num-dirs == rows × cols × layers`. مثلاً شبکه‌ی ۴×۴×۲
> یعنی ۳۲ گره، پس `--num-cpus=32 --num-dirs=32 --mesh-rows=4 --mesh-layers=2`.

## ۷) نتیجه‌ها چطور اندازه‌گیری می‌شوند

- **NetworkInterface** بسته‌ها را تزریق می‌کند (به هر فلیت زمان می‌زند) و موقع رسیدن، تأخیرِ
  شبکه و تأخیرِ صف را اندازه می‌گیرد و جمع می‌زند.
- **GarnetNetwork** جمع‌ها را نگه می‌دارد و چارچوبِ آماریِ gem5 موقعِ نوشتن، آن‌ها را به میانگین
  تبدیل می‌کند: `average_packet_latency = network + queueing`.
- خروجی توی `m5out/stats.txt` می‌رود (هر اجرا بازنویسی می‌شود) و در این پروژه یک
  `GarnetStatsExporter` هم یک رکوردِ JSON به `garnet_results.json` اضافه می‌کند.

## ۸) حلقه‌ی یادگیریِ تقویتی (فقط DeepNR3D و Proposed)

این دو الگوریتمِ یادگیرنده یک عاملِ پایتونی را با **ZMQ** اضافه می‌کنند:

```
RoutingUnit در gem5 (C++، مشتری ZMQ) ──حالت──►  agent.py (پایتون، سرورِ ZMQ)
                                     ◄──اکشن──
```

- gem5 یک **بردارِ حالت (state vector)** می‌سازد، به‌صورت JSON می‌فرستد و **منتظر می‌ماند**
  تا عامل یک اکشنِ `0..5` (یک جهت) برگرداند.
- **پاداش (reward)** هر پرش در `OutputUnit.cc` با فرمولِ `1 / (queuing_delay + 1)` حساب می‌شود.
- gem5 عمداً موقعِ اکشنِ نامعتبر `fatal()` می‌زند — این یعنی «اپیزود تمام شد» و حلقه‌ی آموزش
  دوباره gem5 را اجرا می‌کند.

> [!NOTE]
> 🔵 برای فهمیدنِ سمتِ gem5 لازم نیست یادگیریِ‌ماشین بلد باشی — آن‌جا فقط حالت را سریالایز
> می‌کند و پورتی که برگشت را اعمال می‌کند. کلِ «یادگیری» داخلِ عامل‌های پایتونی است.
